変更されたファイルのセキュリティ監査を実施し、日本語でレポートを出力します。

## 実行手順

### Step 1: 監査対象の特定
```bash
git diff --name-only HEAD~1    # 直近コミットの変更ファイル
git diff --name-only --staged  # ステージング中の変更ファイル
```

### Step 2: 各ファイルを以下の観点で確認

**バックエンド (src/, main.py)**:
- [ ] SQLインジェクション: f文字列や.format()を使った生SQLがないか
- [ ] 認証・認可: APIエンドポイントにアクセス制御が適切にあるか
- [ ] 個人情報のログ出力: 金融データ・認証情報がログに漏れていないか
- [ ] 環境変数: APIキー・パスワードがハードコードされていないか
- [ ] ファイルパス: パストラバーサルのリスクがないか

**フロントエンド (frontend/src/)**:
- [ ] XSS: dangerouslySetInnerHTML の使用、未サニタイズのユーザ入力
- [ ] 認証トークン: localStorageへの機密情報保存
- [ ] APIエンドポイント: 本番URLのハードコード

**設定ファイル・スクリプト**:
- [ ] シークレット漏洩: .env ファイルが誤ってコミットされていないか
- [ ] 依存関係: 新たに追加したパッケージに既知の脆弱性がないか

### Step 3: レポート出力

以下の形式で日本語レポートを出力する：

```
## セキュリティレビューレポート

**判定**: PASS ✅ / REJECT ❌

**監査対象ファイル**:
- `src/xxx.py`
- `frontend/src/xxx.tsx`

**確認項目**:
| 項目 | 結果 |
|---|---|
| SQLインジェクション | ✅ 問題なし |
| XSS | ✅ 問題なし |
| 認証・認可 | ✅ 問題なし |
| 個人情報漏洩 | ✅ 問題なし |
| ハードコードシークレット | ✅ 問題なし |

**発見事項**:
（問題がある場合のみ、修正案とともに記載）
```

REJECT の場合は開発に差し戻し、修正後に再監査を依頼する。
