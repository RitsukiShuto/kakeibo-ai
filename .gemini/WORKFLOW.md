# kakeibo-ai エージェントチーム・運用マニュアル

## 1. チーム・オーケストレーション (Lead Orchestrator)
メインエージェントは「リード・オーケストレーター」として動作し、`task-manager` と連携してタスクを管理します。

1.  **計画策定 (Planning)**: `@task-manager` に依頼し、要求を具体的なステップと `PROJECT_BACKLOG.md` への反映を行う。
2.  **アサイン (Assignment)**: 策定された計画に基づき、各専門エージェント（`@backend-engineer` 等）に順次依頼。
3.  **進捗記録 (Tracking)**: 各ステップ完了時に `@task-manager` に報告し、プロジェクトの状態を更新。
4.  **検証と統合 (Validation)**: `qa-tester` による最終検証後、`task-manager` がタスクの完了を承認する。

## 2. サブエージェント別・作業フロー (SOP)

### A. backend-engineer (TDD実装フロー)
1.  **影響調査**: 既存の `src/models.py` や `src/api` への影響を確認。
2.  **テスト設計 (TDD)**: 実装前に、期待される動作を定義するテストコード（`tests/` 配下）を作成・更新。
3.  **失敗の確認**: テストを実行し、意図通りに失敗することを確認。
4.  **ロジック実装**: テストをパスするための最小限の実装を行う。
5.  **仕様整理とテスト修正**: **ロジック実装完了時に、仕様を再整理した上で単体テスト設計を修正・最適化する。**
6.  **セキュリティレビュー**: 実装完了後、`@security-reviewer` にレビューを依頼。

### B. frontend-engineer (実装フロー)
1.  **コンポーネント調査**: `frontend/src/components` の既存部品を確認。
2.  **UI実装**: React/TypeScript での実装。型安全性を維持。
3.  **ビルド確認**: `npm run build` を実行。
4.  **セキュリティレビュー**: 実装完了後、`@security-reviewer` にレビューを依頼。

### C. qa-tester (TDD検証フロー)
1.  **テスト設計の継続的見直し**: **行った実装に基づき、都度テスト設計を見直し、テストケースを追加・更新する。**
2.  **リグレッション実行**: `python tools/run_regression.py` を実行。
3.  **報告**: 検証結果を詳細に報告。

### D. security-reviewer (セキュリティレビューフロー)
1.  **コード監査**: 実装されたコードに対し、脆弱性や情報漏洩のリスクがないか監査。
2.  **改善提案**: セキュリティ上の懸念点がある場合、具体的な修正方法と共に報告。
3.  **承認**: 安全性が確認された場合、次のフェーズ（統合・デプロイ）への移行を承認。

### D. sre-deploy (展開フロー)
1.  **環境確認**: `docker compose ps` 等で現在の稼働状況を確認。
2.  **構成変更**: `docker-compose.yml` や `infra/` 配下を慎重に修正。
3.  **デプロイ試行**: `sync_to_pi.sh` (ドライラン) またはローカルビルドで安全性を確認。

### E. issue-proposer (提案フロー)
1.  **状況分析**: `PROJECT_BACKLOG.md` と最新のコード・Issueを比較。
2.  **起票**: 実装可能な粒度で新しい Issue (Markdown形式) を提案。

## 3. 共通の禁止事項
- **PR-ONLY**: `main` ブランチへの直接変更は禁止。必ずブランチ作成とPR提案を行う。
- **Quality First**: `run_regression.py` がパスしないコードの提出は禁止。
