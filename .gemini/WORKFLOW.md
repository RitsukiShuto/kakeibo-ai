# kakeibo-ai エージェントチーム・運用マニュアル

## 1. チーム・オーケストレーションと全体ワークフロー
本プロジェクトでは、PMである `task-manager` と、メタ管理者である `agent-manager` の2つの管理レイヤーによる高度なオーケストレーションを採用しています。

**【基本ワークフロー（開発サイクル）】**
`Human` ⇄ `task-manager` ➔ `agent-manager` ➔ `専門エージェント（※）` ➔ `security-reviewer` ➔ `qa-tester` ➔ `task-manager` ➔ `sre-deploy` ➔ `task-manager` ⇄ `Human`

※AI分析やプロンプト変更が絡むタスクの場合、開発の前に `data-analyst` が先行して検証を行います。

1. **計画と承認**: `task-manager` がタスクを分解し、**日本語で**実行計画を策定。Humanの承認を得てから次へ進みます。
2. **差配（ディスパッチ）**: 承認された計画を `task-manager` から `agent-manager` へ移譲。`agent-manager` が最適な専門エージェントへタスクを割り振ります。
3. **分析・プロンプト最適化（必要時）**: AIロジックや家計データの分析が必要な場合、`data-analyst` がプロンプトの検証等を行い、結果を `backend-engineer` に渡します。
4. **実装**: `backend-engineer` や `frontend-engineer` がコードを実装します。
5. **セキュリティ監査**: 実装完了後、`security-reviewer` がコードの安全性を検証します。
6. **品質検証**: 安全確認後、`qa-tester` がリグレッションテスト等を実行し、品質を担保します。
7. **進捗更新＆デプロイ**: テスト通過後、`task-manager` がバックログを更新し、`sre-deploy` が本番環境（Raspberry Pi）へデプロイします。完了後、`task-manager` からHumanへ日本語で完了報告を行います。

## 2. サブエージェント別・作業フロー (SOP)

### A. task-manager (PM・司令塔)
1. 依頼を受けたら、**必ず日本語で**具体的な実行計画（ステップ）を作成する。
2. 計画をHumanに提示し、承認を得る。
3. 承認後、計画を `@agent-manager` に渡してエージェントの差配を依頼する。

### B. agent-manager (マルチエージェント・オーケストレーター)
1. `task-manager` から受け取った計画に基づき、各専門エージェントを自律的に動かす。
2. エージェント間でのハンドオフ（移譲）がスムーズに行われているか監視し、エラーやスタックが発生した場合は調停を行う。

### C. data-analyst (データ分析・プロンプトエンジニアリング)
1. 家計データの分析、または Gemini API のプロンプトテンプレートの最適化を行う。
2. 検証・実験が完了したら、仕様とプロンプトを `@backend-engineer` に移譲して実装を依頼する。

### D. backend-engineer (バックエンド実装フロー)
1. 既存の `src/` 配下の構造を尊重し、テスト（TDD）を作成した上でロジックを実装。
2. 仕様の再整理とテストの最適化を行う。
3. 完了後、絶対に `@security-reviewer` にコード監査を依頼する。

### E. frontend-engineer (フロントエンド実装フロー)
1. React/TypeScript を用い、厳格な型チェックのもとUIを実装。
2. `npm run build` を実行し、エラーがないか検証。
3. 完了後、絶対に `@security-reviewer` にコード監査を依頼する。

### F. security-reviewer (セキュリティ監査フロー)
1. コードに対し、脆弱性や情報漏洩のリスクを監査。問題があれば修正案とともに差し戻す。
2. 安全性が確認されたら、`@qa-tester` に移譲して最終的な品質チェックを依頼する。

### G. qa-tester (品質保証・テストフロー)
1. 実装に基づきテストケースを追加・更新し、`python tools/run_regression.py` を実行。
2. 品質が担保されたら `@task-manager` に移譲し、進捗更新を依頼する。

### H. sre-deploy (展開・インフラフロー)
1. Docker環境や本番（Raspberry Pi）の状況を確認し、安全にデプロイを実行。
2. 完了後、`@task-manager` に移譲して最終クローズを依頼する。

### I. issue-proposer (提案・分析フロー)
1. バックログやコードベースを分析し、新しい機能や改善案をMarkdown形式で提案。
2. 提案をまとめたら `@task-manager` に移譲し、計画策定を依頼する。

## 3. 共通の禁止事項
- **非日本語での対話**: `@task-manager` は、Humanとのやり取りや計画策定において英語を使用してはならない。必ず日本語で出力すること。
- **承認なしの進行**: 実行計画についてHumanの事前承認を得ずに開発フェーズへ移行してはならない。
- **PR-ONLY**: `main` ブランチへの直接変更は禁止。必ずブランチ作成とPR提案を行う。
