## 🚨 致命的なバグ (Critical)

### F-1. budget.json の新旧フォーマット不整合（予実管理が完全に壊れている）

**テスト結果:** 予実管理セクションが「その他: 0 / 206,220円」の1行しか表示されない。

**原因の特定:**
- `budget.json` が **旧フォーマット** (`categories` キー直下にカテゴリ一覧) で保存されている
- API の `load_budget()` 関数には「旧形式の互換性維持」コードがあるが、旧形式を新形式に変換する際に **すべてのカテゴリを「その他」の下に強制マッピング** してしまう:

```python
# src/api/main.py L50-53
if "budget" not in data.get("monthly", {}):
    old_categories = data.get("monthly", {}).get("categories", {})
    if "monthly" not in data: data["monthly"] = {}
    data["monthly"]["budget"] = {"variable": {"その他": old_categories}}  # ← 全部「その他」に
```

- そして `get_budget_category_totals()` は **大項目（この場合「その他」）ごとに合計** するため、11カテゴリすべてが合算されて1行になる

**影響:** ダッシュボードの最も重要な機能がまったく機能していない。

**修正案:** `load_budget()` の互換性ロジックを修正し、旧フォーマットのカテゴリをそれぞれ独立した大項目として扱う。

---

### F-2. 「かんたん設定」と「JSON設定」のデータ不整合で設定が壊れる

**テスト結果:**
1. 「JSON設定」で `budget.json` を編集して保存
2. 保存時に不正なJSONだとエラー: `"Unexpected non-whitespace character after JSON at position 311"`
3. 「かんたん設定」→「予算設定」タブでは「カテゴリ別予算 (固定費)」「カテゴリ別予算 (変動費)」がどちらも **空**

**原因:** `budget.json` が旧形式（`categories`フラット構造）なので、「かんたん設定」UIが想定する新形式（`budget.fixed` / `budget.variable` 構造）と一致しない。UIで保存すると新形式で上書きされ、旧形式のカテゴリ情報が失われる。

> [!CAUTION]
> 「かんたん設定」で保存すると `budget.json` が旧→新形式に書き換わり、**カテゴリ予算データが消失する**。これは非常に危険。

## 🔴 重大なバグ (High)

### F-3. KPI値がタイムフレーム切替で変わらない

**テスト結果:** 日次→週次→月次のタブを切り替えても、KPI（予算・実支出・残額・総資産）の値が **同一のまま**。

**原因:** API `/api/kpi` に `timeframe` パラメータが渡されておらず、常に `current_month` の全期間で集計。`Dashboard.tsx` の `fetchData()` でも `timeframe` を KPI 取得時に利用していない。

### F-4. AIレビュー詳細のレポートファイル読み込み失敗

**テスト結果:** 月次分析レポートの詳細を開くと「**レポートの読み込みに失敗しました。ファイルが存在しない可能性があります。**」

**原因:** `analysis_history` テーブルの `report_path` がObsidianの保存先パスを含んでおり、APIサーバーの実行環境からそのパスが参照できない。日次レポートは読み込めている（パスが `./reports/` 相対パス）ので、月次レポートのパス設定の問題。

### F-5. 明細検索がリアルタイムではなくEnterキー送信後に実行

**テスト結果:** 検索ボックスにテキストを入力しただけではフィルタリングされず、**Enterキーを押すかフォーカスを外した後にAPIリクエストが発生**。これ自体は正常な動作だが、UXとしてはリアルタイム検索か「検索ボタン」の追加が望ましい。プレースホルダー「明細を検索 (内容、カテゴリ、金額など)...」の表記から、ユーザーはリアルタイム検索を期待する。

### F-6. 明細一覧にページネーションがない

**テスト結果:** `limit=50` で固定取得。スクロールしても新しいデータがロードされない。50件以上の取引がある場合、古いデータにアクセスできない。

### F-7. `genre`（中項目）の更新がAPIで無視される

**テスト結果（コード分析）:** フロントエンドでは中項目ドロップダウンの編集UIがあるが、`TransactionUpdate` Pydanticモデルに `genre` フィールドがない。保存してもサイレントに無視される。

## 🏗️ アーキテクチャ・セキュリティの課題

### F-21. SQLインジェクションのリスク（🚨 緊急）
API内でf-stringによるSQL構築（L172, L203）。パラメータ化クエリに統一すべき。

### F-22. DB接続の例外安全性
各エンドポイントで `conn.close()` が `try`/`finally` なしで呼ばれており、例外時にリークする。

### F-23. CORS全開（`allow_origins=["*"]`）
本番デプロイ前に適切なオリジンに制限すること。

### F-24. 設定APIのバリデーション不足
`PUT /api/settings/budget` が任意のJSONを受け入れてファイルに書き込む。Pydanticモデルでの検証が必要。

### F-25. `any`型の多用（TypeScript）
`Dashboard.tsx`, `ExpenseSplitter.tsx`, `Settings.tsx` で `useState<any>` が多数。型安全性が失われている。
